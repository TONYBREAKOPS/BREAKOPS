2. ShiftOps

File: shiftops.html~9,500 linesDaily Operations

Overview

ShiftOps is the primary daily operations tool. It manages employee scheduling across day and night shifts, tracks attendance (present, absent, tardy, partial, trainer, trainee), assigns seat positions, monitors staffing adequacy, handles overtime imports, generates timesheets, and produces printable/exportable reports.

Header Controls

  • Pay Period (PP) — Select from PP3 onward (14-day cycles starting 2026-01-10)
  • Week — Week 1 or Week 2 within the pay period
  • Day — Saturday through Friday with the specific date shown
  • Shift — Day Shift (0600–1800) or Night Shift (1800–0600). Auto-selects based on current time
  • Panel View — "All 3 blocks" or individual 4-hour blocks, plus an employee search field

Options Menu

  • Import OT — Import Microsoft Teams overtime schedule (XLSX or CSV)
  • Save XLSX — Export current day's roster to a styled Excel workbook
  • Timesheet — Generate unofficial timesheets for the full 14-day pay period
  • Print Daily — Portrait letter-size PDF (roster + comments)
  • Print Weekly — 8-page legal-size PDF (regulars + OT for both shifts)
  • Download Weekly Excel — Weekly Excel workbook with staffing data
  • Print Excel Legend — Reference guide for weekly Excel color codes
  • Shift Swap — Swap schedules between two employees
  • Push to BreakOPS — Push staffing data to the BreakOPS system

Shift Board Layout

Each shift has three 4-hour blocks displayed as columns. Each column contains stacked cards:

  1. Staffing Numbers — Hour-by-hour required vs. scheduled headcount (SHORT / STAFFED / OVER)
  2. Calltakers Table — Name, Role, Seat, Worked, Hrs, Present, Comment
  3. Dispatchers Table — Same format
  4. PIC Table — Same format
  5. OTHER Table — For OTH roles (SD, TRN, B, ADM)
  6. Supervisors Table — Same format
  7. Absent Table — Shows absent/tardy/partial with Reason column
  8. OT / Extra Bar — Add overtime employees with Role and Segment selectors

3. BreakOPS — Admin Panel

File: admin.html~4,300 linesBreak Scheduling

Overview

BreakOPS Admin is a real-time break scheduling system for managing employee breaks across four positions (Calltaker, Dispatcher, PIC, ANET). It uses a turn-based sequential system where each employee picks their break in order, with skip timers and second-chance mechanics.

Break Picker

  • DEPLOY BREAKS — Starts break selection for all positions (must lock editing first)
  • PAUSE ALL — Pauses all timers
  • Timer Duration — 10sec to 10min (default 5 min)
  • Slot Capacity Manager — Adjust per-slot, per-position capacity (CT=3, DP=2, PIC=1, ANET=1)
  • NEXT UP (yellow) — Current picker's turn
  • SKIPPED (gray) — Timer expired, waiting for second chance
  • 2ND CHANCE (orange) — Skipped person getting another turn
  • Pre-Selections & Auto-Pick — Pre-set preferences for automatic assignment

Additional Features

  • Dispatcher channel assignment cards (10+ channels)
  • Position lineup tables with override dropdowns
  • Add extra people (overtime/fill-ins)
  • NOT IN STAFFING LISTS panel for absent employees
  • Print break sheets (2-page: CT breaks + DP/PIC breaks with relief table)
  • Push seats to ShiftOps
  • Coverage mini-tables per position

Firebase

Uses two Firebase projects: legitimous-2f2d2 (breaks, state, config) and commandops-93846 (ShiftOps push only).

6. RotationOps

File: rotationopsv2.html~14,500 linesMonthly Rotation

Overview

RotationOps generates monthly rotation schedules assigning dispatchers to 10 radio channels (North, East, South, West, Northwest, Central, DC1, DC2, Relief 1, Relief 2) across six 4-hour blocks per day. Uses AI-powered fairness optimization. Works alongside the Dispatch Board (Section 7) which provides the live daily view.

Core Functions

  • RANDOMIZE — Runs hundreds of full-month attempts (~60 sec), keeps the best-scoring plan
  • BALANCE — Post-optimization pass that swaps to improve fairness
  • CHECK VIOLATIONS — Scans for rule violations (same-channel-twice, weekly caps)
  • CLEAR ASSIGNMENTS — Password-protected, wipes all assignments

View Tabs

  • Day — Single-day, click names for SWAP/MOVE/MARK ABSENT, click NEEDED to assign
  • Week — 7-day grid with search
  • Month — Full-month calendar
  • Roster — Per-employee grid split by Day/Night/Split shift, printable
  • Monthly Exceptions — Vacations, exclusions, trainers with block/channel overrides

Export Options

  • Export Totals (fairness stats) / Export Excel / Export CSV
  • Export Open Shifts (Teams-compatible .xlsx)
  • Push to ShiftOps

Configuration

10 channels + 7th channel (Fri/Sat nights). Six 4-hour blocks. Weekly caps: CT 2, Relief 2, DC 2, Primary 2. SD monthly cap: 32 blocks. Dark/Light theme.

7. Dispatch Board

File: dispatch.htmlLive OperationsReal-time Sync

Overview

The Dispatch Board is a live, real-time rotation board that displays current channel assignments for dispatchers across all 10 radio channels plus calltaker slots. It serves as the daily companion to RotationOps (Section 6), which handles monthly planning.

Board Layout

  • 10 Dispatch Channels — North, East, South, West, Northwest, Central, DC1, DC2, Relief 1, Relief 2 — each with a color-coded left border
  • 3 Calltaker Slots — Per time block
  • 3 Time Blocks — Day Shift (6A–10A, 10A–2P, 2P–6P) or Night Shift (6P–10P, 10P–2A, 2A–6A)
  • Day Selector — Sunday through Saturday tabs

Controls

  • Power Button — ONLINE / OFFLINE toggle. When offline, all viewers see "PENDING" in every cell
  • Lock / Unlock — Password-protected editing. Auto-locks after 3 minutes of inactivity
  • Paste Block — Bulk-paste all 10 channel names at once (one per line)
  • Save — Pushes current board state to Firebase for all connected devices
  • Clear Block — Reset all assignments for the current shift

Name Hover Trail

Hovering over a dispatcher's name highlights all blocks where that person appears across the shift, with animated SVG connector lines and arrows showing their rotation path between blocks.

Mobile View

Fully responsive — compact grid with abbreviated channel labels (N, E, S, W, NW, C, DC1, DC2, R1, R2, CT), optimized touch targets, and tap-to-trail on locked boards. Fills the full viewport with no scrolling required.

Auto Shift Switching

Automatically switches between Day and Night shift tabs based on current time (Day before 5 PM, Night after 5 PM). Manual selection overrides auto-switching for the session.

Announcements

A text area below the board for supervisor announcements. Syncs in real-time to all connected devices. Editable only when unlocked.

Firebase

Uses legitimous-2f2d2 — stores board data, announcement text, and power state in appState/rotationBoard. Real-time sync via onSnapshot. Falls back to localStorage when offline.

9. StaffingOps

File: staffingopsv2.htmlStaffing Analysis & OT Distribution

Three Main Views

STAFFING NUMBERS

7-column (Mon–Sun) x 12-row grid split into DAY and NIGHT shifts. Each cell shows current/needed, color-coded: SHORT ADEQUATE OVER. Hover to see names.

OT NEEDS (Monthly Calendar)

Full-month horizontal scrolling table with 6 blocks per day (6A-10A, 10A-2P, 2P-6P, 6P-10P, 10P-2A, 2A-6A). Summary bar at top shows total OT slots and hours needed. Rows include:

  • STAFF row — Numbered chips for each scheduled employee. Blue DISP IN CT badges for dispatchers assigned to calltaking via RotationOps. Red OT badges for overtime assignments. Red pulsing DP NEEDED / CT NEEDED / PIC NEEDED badges for unfilled slots. Block footer shows NEED X (red), FULL STAFFED (green), or OVER +X (purple)
  • OT BIDS row — Shows who bid per block: green (PREFER), yellow (CAN), red with OT badge (assigned). Only employees qualified for the current role appear. Click a NEEDED badge to manually assign OT
  • VACATION section — Cyan names at the bottom of each OT BIDS cell showing who is on vacation that day
  • SPECIAL DETAIL row (Dispatcher only) — Purple SD row showing predicted special detail assignments per block

OT Distribution (Run Distribution)

Automated overtime assignment engine. It reads employee bids from OT Bidding (Section 10) and fills only the slots where staffing is actually short.

Admin Bid Panel

  • Submitted list — Shows every employee who submitted a bid for the current period
  • Click any name to expand and see their slot-by-slot bids grouped by date (green = PREFER, yellow = CAN)
  • R1 / R2 hours — Shows what each employee requested

Distribution Algorithm

Pre-Processing

  • Needed Slots Only — Ignores any block where staffing is already adequate
  • Cross-Role Check — Pulls OT assignments from all three roles (Dispatcher, PIC, Calltaker) so nobody gets double-booked
  • Shift Conflict Check — Won't assign OT during blocks someone is already working their regular shift
  • Vacation Exclusion — Anyone on vacation that day is automatically skipped
  • Multi-Fill — If a slot needs 3 people, it assigns up to 3

Round 1 — Scarcity-First

  1. Finds the hardest-to-fill slot (fewest eligible candidates)
  2. Picks the best candidate: fewest options → "prefer" over "can" → highest seniority
  3. Assigns one slot, then re-checks everything and repeats
  4. Each employee capped at their desiredR1 hours per week (default 4)

Round 2 — Opt-In Extra Hours

  1. Only employees who set desiredR2 > 0 are eligible — this round is opt-in
  2. Weekly cap becomes desiredR1 + desiredR2
  3. Same scarcity-first logic, same tiebreakers

Safety: The algorithm has a hard cap of 5,000 iterations and yields to the browser every 20 cycles so the page never freezes.

Run All Roles

One-click button that runs distribution for all three roles in order: Dispatcher → PIC → Calltaker. Each role's results feed into the next so cross-role double-booking is prevented automatically. Shows a combined results screen at the end.

Distribution Results

  • Fill rate — Percentage of needed slots that got filled
  • R1 / R2 badges — Each assignment shows which round assigned it
  • Unfilled slots — Listed in red with how many candidates were available
  • Publish — Saves approved assignments to Firestore

Clear OT

Wipes all OT assignments for the current role and period. Use this to start fresh before re-running distribution.

Import / Export

  • Export — Copies all current staffing data (schedules, OT assignments, bids) to clipboard as JSON
  • Import — Paste previously exported JSON to restore a saved state

Cross-Position Workflow

Designed for a three-pass workflow:

  1. Dispatcher first — Only dispatchers and PIC-qualified people can fill dispatch OT
  2. PIC next — Fill remaining PIC slots with PIC-qualified people
  3. Calltaker last — Everyone can work calltaking, so fill these last

The system checks all three roles before each run so no one gets double-booked.

Hypothetical Schedule Modeling

  • Dual tabs (Set A / Set B) with renamable labels
  • Load employees or add blank entries, click day pills for custom IN/OUT times
  • APPLY to see grid impact with +/- change badges
  • LOAD RECS / TRIM / SUGGEST for AI-assisted optimization
  • Print before/after comparison reports

10. OT Bidding

File: otbidding.htmlEmployee-FacingOT Bid Submission

Overview

OT Bidding is the employee-facing companion to StaffingOps (Section 9). Employees log in and submit their overtime availability for the current bid period. The admin then runs distribution in StaffingOps to auto-assign OT based on these bids.

Bid Submission

  • Grid Layout — Days as columns, 4-hour blocks as rows (6A–10A, 10A–2P, 2P–6P, 6P–10P, 10P–2A, 2A–6A)
  • Four Cell States — Click to cycle: PREFER (first choice), CAN (available but not preferred), CAN'T (unavailable), or empty (no preference set)
  • Scheduled Blocks — Blocks you're already working are greyed out
  • Vacation Days — Days you're on vacation are marked and excluded from assignment
  • PIC Role Badges — If you're PIC-qualified, a purple PIC badge shows on your bid card
  • Desired Hours — Set your Round 1 hours (default 4). Round 2 is opt-in — set R2 hours only if you want extra OT beyond R1

How Distribution Uses Bids

  • OT is only assigned where there's an actual staffing need
  • Round 1 fills the hardest slots first, capped at your R1 hours per week
  • Round 2 is opt-in only — you must set R2 > 0 to be included. Cap becomes R1 + R2
  • Marking more slots improves your chances — the algorithm favors flexibility
  • You won't be assigned during your regular shift or on vacation days
  • Cross-role tracking prevents double-booking (e.g., Dispatch + Calltaking same block)

Firebase

Bids are stored in the ot_bids collection using subcollections:

  • ot_bids/{period}/emp_bids/{employeeName} — Individual bid documents (slots, desired hours, timestamp)
  • ot_bids/{period}/results/approved — Published OT assignments after distribution

Period format example: DISPATCHER-2026-03, CALLTAKER-2026-03, PIC-2026-03.

12. MandateOps

File: mandateops.htmlOvertime Mandate Ranking

Overview

Generates ranked lists of employees eligible for mandatory overtime, records acceptance/refusal, and produces formal refusal forms.

Ranking Algorithm

  1. Shortest work day (ascending)
  2. Least OT hours in last 7 days (ascending)
  3. Highest seniority number (descending) for ties

Eligibility Rules

  • Shift must end when mandate block starts
  • Must be Present
  • No existing OT overlap
  • Would not exceed 16-hour cap

Actions

Mandate (accept) / Refuse (decline) / Undo (today only) / Print Refusal (formal form with signature lines).

Cross-Qualification

CT: check CT → DP → PIC. DP: check DP → PIC. PIC: PIC only.

16. Cross-App Architecture

Two Firebase Projects

ProjectApps
commandops-93846ShiftOps, RotationOps, StaffingOps, OT Bidding, AttendanceOps, ScheduleOps, MandateOps, DecisionOps, Employee DB, Bug Reports
legitimous-2f2d2BreakOPS (Admin, Staff View, Backup), Dispatch Board

Shared Data Flow

Employee Database Editor (employees.html)
    |
    +-- employees collection ----> ALL APPS (roster data)
    +-- *_Adequate_Staffing -----> ShiftOps, StaffingOps, ScheduleOps, BreakOPS
    +-- *_Seat_Assignments ------> ShiftOps, BreakOPS
    +-- Absent_Reason -----------> ShiftOps
    +-- scheduledChanges --------> ShiftOps, RotationOps
    +-- Export Schedules --------> BreakOPS (manual copy-paste of BASE array)

RotationOps (rotationopsv2.html)
    |
    +-- rotation_plans ----------> StaffingOps (vacation/CT data)
    |                           -> AttendanceOps (vacation/training)
    |                           -> ShiftOps (vacation/training auto-marks)
    +-- daily_edits <----------- ShiftOps (push rotation to shifts)

Dispatch Board (dispatch.html)
    |
    +-- appState/rotationBoard --> Real-time board state (all connected viewers)

ShiftOps (shiftops.html)
    |
    +-- daily_edits -------------> AttendanceOps (absence extraction)
    |                           -> MandateOps (attendance checks)
    |                           -> StaffingOps (OT tracking)
    +-- BreakOPS push -----------> BreakOPS (position overrides, OT extras)

OT Bidding (otbidding.html)
    |
    +-- ot_bids/{period}/emp_bids --> StaffingOps (bid data for distribution)

StaffingOps (staffingopsv2.html)
    |
    +-- ot_bids/{period}/results --> Published OT assignments (per role)
    +-- reads all 3 role periods --> Prevents double-booking across positions

AttendanceOps (vacationops.html)
    |
    +-- rotation_plans ----------> RotationOps, StaffingOps, ShiftOps

All Apps --> bugReports ----------> Bug Reports Dashboard (bugs.html)

Common Patterns

  • Single-file HTML — All apps self-contained with inline CSS and JS
  • Dark/Light Theme — Consistent toggle with CSS custom properties
  • Firebase Firestore — Real-time shared state via onSnapshot listeners
  • Toast Notifications — Transient success/error messages
  • Responsive Design — CSS Grid and Flexbox
  • Print Stylesheets — @media print rules for formatted output
  • Bug Reporting — Every app has a "Report Bug" button
  • Firebase Auth — Email/password login (except Hub and BreakOPS Staff)
  • Auto-Save — Debounced (1.5–3 sec) with echo-loop guards
